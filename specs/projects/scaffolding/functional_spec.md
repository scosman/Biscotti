# Functional Spec: Project 0 — Scaffolding & Tooling

## Purpose & "users"

This Project ships **no product behavior**. Its "users" are **developers and CI** (human and agent). So the "features" are the **developer command surface**, the **CI gates**, and the **repo skeleton** that every later Project builds inside. The quality bar: a fresh checkout can build, test, lint, and launch an empty app with one set of commands that behave identically locally and in CI.

It realizes the [`architecture.md` workspace layout](../../../architecture.md#workspace-layout) and the [thin-app composition rule](../../../architecture.md#thin-app-composition): everything testable lives in SPM packages (`swift build`/`swift test`); the only thing needing `xcodebuild` is a logic-free app shell.

Locked decisions from the overview: **XcodeGen** for the app project, bundle ID **`net.scosman.biscotti`**, **bare-window** shell.

## 2. Command surface (Makefile — the contract)

One source of truth for both humans and CI. CI calls these same targets, so "passes locally" ⇒ "passes in CI".

| Target | Does | Gating? |
|---|---|---|
| `make bootstrap` | Install dev tools via Homebrew (`brew bundle` from `Brewfile`); print guidance if Homebrew absent. Idempotent. | — |
| `make generate` | `xcodegen generate` in `App/` → produces `Biscotti.xcodeproj`. | — |
| `make build` | `swift build` all packages. (App build lives in `make build-app`.) | — |
| `make test` | `swift test` across packages. **The gating tier.** | ✅ gates |
| `make lint` | Check only: `swiftformat --lint .` then `swiftlint --strict`. Non-mutating; non-zero exit on any issue. | ✅ gates |
| `make format` | Mutating autofix: `swiftformat .` then `swiftlint --fix`. (SwiftFormat first, then SwiftLint autocorrect — see §4.) | — |
| `make build-app` | `make generate` then `xcodebuild build` the app (ad-hoc signed). **Non-gating app tier.** | ⚠️ non-gating |
| `make test-app` | `xcodebuild test` the app/UI test scheme (empty for now). **Non-gating.** | ⚠️ non-gating |
| `make hooks` | Opt-in: `git config core.hooksPath .githooks` to enable the pre-commit hook. | — |
| `make clean` | Remove `.build/`, `DerivedData/`, generated `.xcodeproj`. | — |
| `make ci` | What the gating CI job runs: `lint` + `test` (+ `build`). | — |

**Why a Makefile + Homebrew:** the user asked that local and CI share commands — the Makefile is that single source of truth, invoked identically by humans, CI, and the agent MCP (§9). Dev tools (xcodegen, swiftlint, swiftformat) are installed via a `Brewfile` (`brew bundle`); CI installs the same way. *(Version pinning is intentionally light — Homebrew tracks latest. A SwiftFormat/SwiftLint upgrade can in principle reformat code and redden CI with no source change; if that bites, pin the affected formulae in the `Brewfile` then. Considered Mint for exact pins but chose Homebrew for simplicity per the author's call.)*

## 3. Build settings: strict concurrency + warnings-as-errors

The author's listed Xcode settings are honored, and their SPM equivalents are applied so the **package tier** (where 99% of code lives) is held to the same bar.

**SPM packages (`Package.swift`):**
- `swiftLanguageModes: [.v6]` — Swift 6 language mode, which makes **complete** concurrency checking the default (the SPM equivalent of `SWIFT_STRICT_CONCURRENCY=complete`).
- Per-target `swiftSettings: [.treatAllWarnings(as: .error)]` (supported SwiftPM API; **not** `unsafeFlags`, which would forbid the package being consumed as a dependency).
- `// swift-tools-version:6.2` (minimum for `treatAllWarnings(as:)`), `platforms: [.macOS(.v15)]`.

**Xcode app (`project.yml` settings):**
```yaml
SWIFT_VERSION: "6.0"
SWIFT_STRICT_CONCURRENCY: complete
SWIFT_TREAT_WARNINGS_AS_ERRORS: YES
CLANG_TREAT_WARNINGS_AS_ERRORS: YES
GCC_TREAT_WARNINGS_AS_ERRORS: YES
OTHER_SWIFT_FLAGS: "$(inherited) -warnings-as-errors"
MACOSX_DEPLOYMENT_TARGET: "15.0"
```
*(`SWIFT_STRICT_CONCURRENCY=complete` is redundant under Swift 6 language mode but kept explicit, as the author specified, and to remain correct if a target stays on the Swift 5 mode.)*

**Consequence:** the skeleton itself must compile clean under these — no warnings, no concurrency diagnostics. The placeholder code is written to satisfy them.

## 4. Lint & format (SwiftLint + SwiftFormat, no turf war)

Both tools are used, with a clean division so they don't fight:

- **SwiftFormat owns formatting** (whitespace, wrapping, imports ordering, etc.). `.swiftformat` pins `--swiftversion 6.0` and a deliberate rule set; this is the autoritative formatter.
- **SwiftLint owns semantic/style lint** (naming, complexity, smell rules), **not** formatting. In `.swiftlint.yml`, disable rules that overlap with SwiftFormat's formatting so the two can't disagree.
- **Stricter ruleset (author's ask):** enable a curated set of `opt_in_rules` (e.g. `force_unwrapping`, `empty_count`, `first_where`, `closure_spacing`, `redundant_type_annotation`, `unused_declaration`, …) and run `swiftlint --strict` so **warnings fail** the lint. (Final rule list decided in implementation; the principle is "opt into the strict rules, treat warnings as errors.")
- **Ordering:** `make format` and the pre-commit hook run **SwiftFormat first, then `swiftlint --fix`**, so autocorrect operates on already-formatted code and the two converge.
- **Scope:** lint/format cover `Packages/**/Sources`, `Packages/**/Tests`, and `App/Sources` (exclude generated/`.build`/`DerivedData`).

## 7. CI (GitHub Actions, two tiers)

One workflow (`ci.yml`), triggered on PRs and pushes to `main`, on a `macos-15` runner with a pinned Xcode (via `maxim-lobanov/setup-xcode`), dev tools installed via `make bootstrap` (`brew bundle`).

- **Tier 1 — gating package tier (required):** `make ci` → `make lint` + `make test` (+ `make build`). Operates entirely under `swift`; fast, deterministic. **This is the required status check** for merging.
- **Tier 2 — app/UI tier (non-gating):** `make build-app` (xcodegen generate + xcodebuild build, ad-hoc). Reported on the PR but **not** a required check — runs as a separate job marked `continue-on-error` / left out of branch-protection required checks. Rationale: xcodebuild is the historically flaky part; we want its signal without it blocking unrelated package work. Branch-protection config (which checks are required) is a one-time GitHub repo setting, documented in `CLAUDE.md`, not code.

Caching: SPM build cache keyed on `Package.resolved`; Homebrew tool install cached where practical.

## 8. Pre-commit hook (opt-in template)

- Lives at `.githooks/pre-commit` (checked in, version-controlled).
- Runs, in order, blocking the commit on the first failure:
  1. **Format** — `make format` (SwiftFormat then `swiftlint --fix`), and **re-stage** any files it changed so the commit includes the formatted result. *(This auto-restages formatting-only edits — the deliberate choice over a format-*check*, so a developer never has to manually re-run after the hook tidies whitespace.)*
  2. **Lint** — `make lint` (`swiftformat --lint` + `swiftlint --strict`); fails on any remaining issue (i.e. a lint problem autofix couldn't resolve).
  3. **Fast tests** — `swift test` across the packages. **Package tests only — never `xcodebuild`** (the slow/flaky app tier stays out of the commit path; CI Tier 2 covers it).
- On failure: non-zero exit aborts the commit with a message naming the failing step.
- **Opt-in by design** (author's "template you copy"): not active until the developer runs `make hooks` (sets `core.hooksPath .githooks`). Never auto-installed.
- **Speed note:** as the package suite grows this hook gets slower; if it becomes a drag, the escape hatch is to scope tests or move them to a pre-push hook. Acceptable for now — the suite is tiny.

## 9. Agent/build integration (MCP)

Two MCP servers are registered for agents via a checked-in **`.mcp.json`** (Claude Code auto-discovers it at repo root):

- **`hooks_mcp`** ([scosman/hooks_mcp](https://github.com/scosman/hooks_mcp)) — the **primary** agent command surface. A `hooks_mcp.yaml` at repo root exposes our common commands as named, whitelisted actions, so an agent can discover and run them (and read shared prompt files) without guessing raw shell. **Each action wraps the corresponding Makefile target** — the Makefile stays the single source of truth; hooks_mcp is the agent-facing veneer over it. Minimum actions mirror §2: `bootstrap`, `generate`, `build`, `test`, `lint`, `format`, `build-app`, `test-app`. Example shape:
  ```yaml
  server_name: "Biscotti"
  server_description: "Biscotti dev commands (wrap the Makefile)"
  actions:
    - name: "test"
      description: "Run the gating package test suite (swift test)"
      command: "make test"
    - name: "lint"
      description: "Check formatting + lint (non-mutating)"
      command: "make lint"
    - name: "format"
      description: "Auto-format and autofix lint"
      command: "make format"
    - name: "build_app"
      description: "Generate the Xcode project and build the app (non-gating)"
      command: "make build-app"
    # …bootstrap / generate / build / test_app
  ```
  Runtime: `uvx hooks-mcp` (via `uv`); the `.mcp.json` entry runs `uvx hooks-mcp --working-directory .`. `uv` is added to the `Brewfile`.
- **XcodeBuildMCP** — for the interactive `xcodebuild`/run/launch/log-capture operations a `make` target can't model (launch the app, read build settings, capture run logs). Registered alongside hooks_mcp.

Day-to-day agent work still stays in `Packages/*` under `swift build`/`swift test` — no signing, no MCP needed. The MCP layer is for the standard command surface and the rare app-target paths. Both servers are documented in `CLAUDE.md`.

## 11. Edge cases & failure handling (developer-facing)

| Situation | Behavior |
|---|---|
| Dev tool missing (no Homebrew/xcodegen) | Make targets fail fast with a clear message → run `make bootstrap` / install Homebrew. No silent partial runs. |
| Tool version drift local vs CI | Both install from the same `Brewfile` and call the same Makefile targets. (Pins are loose; pin a formula in `Brewfile` if a tool upgrade reformats and reddens CI.) |
| `uv`/hooks_mcp not installed when an agent calls it | hooks_mcp actions just shell out to `make`; absent `uv`, an agent falls back to running the `make` targets directly (or `make bootstrap` adds `uv`). |
| `xcodebuild` flaky / fails in CI | Confined to the **non-gating** Tier 2 job; never blocks merge. Surfaced for visibility. |
| Generated `.xcodeproj` stale or committed by mistake | It's gitignored; `make generate` rebuilds it; `make build-app` regenerates first. |
| Lint and format disagree | Prevented by the SwiftFormat-owns-formatting / SwiftLint-formatting-rules-disabled split (§4). |
| Warnings introduced | Fail the build (packages and app) via warnings-as-errors — caught in Tier 1. |

## 12. Acceptance criteria (definition of done)

1. Fresh clone → `make bootstrap` installs the dev tools via Homebrew.
2. `make test` runs `swift test` green (the one trivial test passes).
3. `make lint` passes clean on the skeleton; `make format` is a no-op on already-clean code.
4. `make build-app` generates the project and `xcodebuild` builds the app, ad-hoc signed.
5. Launching the built app shows an empty placeholder window (manually verified once; recorded).
6. CI: Tier 1 (lint + test + build) green and configured as the **required** check; Tier 2 (app build) runs and reports without gating.
7. `make hooks` installs the pre-commit hook; it formats + re-stages, lints, and runs `swift test`, blocking the commit on any failure.
8. Build settings enforce Swift 6 language mode (complete concurrency) + warnings-as-errors across packages and app.
9. `hooks_mcp.yaml` + `.mcp.json` present; an agent can list the actions and run `test`/`lint`/`format` through hooks_mcp (each wrapping its `make` target).
10. `CLAUDE.md` updated; `research/permissions` bundle-ID line corrected to `net.scosman.biscotti`.
