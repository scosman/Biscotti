# Functional Spec: Custom Summary Prompt

## 1. Summary

Let the user edit the instruction prompt Biscotti's on-device model uses to write
each meeting summary. There is **one** editing surface — a Markdown editor sheet —
reused in two modes:

- **Global (Settings):** edit and **save** the default prompt. Used for all future
  summaries. No back-fill of existing summaries.
- **Per-meeting (one-off):** open the same sheet from a meeting, customize the
  prompt, and **regenerate that one meeting's summary**. Edits are *not* saved by
  default; an optional checkbox can also promote them to the global default.

This is power-user-direct: the user edits the **raw instruction prompt**, not a
builder. There are no template variables, section toggles, or NL-to-prompt layers.

## 2. Concepts & scope

### 2.1 What the prompt is (and isn't)

A summary is generated by sending the on-device model a multi-turn conversation:

1. A fixed **system** instruction.
2. A **first user turn** containing the meeting *context* — `<meeting_details>`
   (title, date, location, conference, invitees, description) and `<transcript>` —
   followed by the **summary instruction**.
   (When speaker identification also runs, the transcript/context arrive in an
   earlier turn and the summary instruction arrives as a follow-up turn.)

**This feature exposes only the summary instruction** — today the constant
`IntelligencePrompts.summaryTaskInstructions`. The context (transcript, title,
date, participants) is assembled automatically by the app and is **not** part of the
editable text.

Therefore:

- The editable prompt is **plain instruction text**. It contains **no template
  variables / handlebars** (`{{transcript}}`, etc.) and must not be presented as
  supporting them. It must not reference "the context above" mechanically — it is
  simply instructions to the model, which already has the transcript in context.
- The prompt affects **only the summary**. Title generation and speaker
  identification keep their own fixed instructions and are out of scope.

### 2.2 The editable prompt is the *whole* summary instruction

Per decision: there is **no hidden scaffolding**. The text the user sees and edits
is exactly the instruction sent to the model for the summary turn. The factory
default **is today's `summaryTaskInstructions` verbatim** (so default behavior is
unchanged):

```
Produce a clear, well-organized markdown summary of the meeting covering the key
decisions, discussion topics, and outcomes. At the end, include a "## Action Items"
section as a checklist using `- [ ]` format, with owners noted when clear from the
transcript. You may reference the speaker names identified earlier, but do NOT
repeat or restate the speaker mapping/list. Your response must contain ONLY a
summary of the meeting and nothing else: no speaker mapping, no preamble, no
commentary, and no content that is not in the transcript. Output markdown only, and
begin your response directly with the summary.
```

> Note: this default keeps its own output-hygiene guardrails (don't restate the
> speaker list, no preamble, markdown only, begin directly). If a user deletes
> those when editing, output quality is their responsibility — this is the
> power-user contract. The example additions (§5) are written to coexist with the
> default's structure.

### 2.3 Out of scope

- Editing the system prompt, the speaker-ID prompt, the title prompt, or the
  context assembly.
- Template variables, prompt "sections"/toggles, a guided builder, or NL-to-prompt.
- Multiple saved prompts / prompt presets / per-meeting persisted prompts.
- Back-filling existing summaries when the global prompt changes.
- Versioning/undo history for summaries beyond what already exists (none).

## 3. Entry points

### 3.1 Settings → AI Enhancements

Add **one row** in the existing **AI Enhancements** section of Settings, placed
between the **AI Analysis & Summary** toggle and the **AI Language Model** row.

| | |
|---|---|
| **Title** | Summary Prompt |
| **Subtitle** | Customize the instructions used to write meeting summaries. |
| **Accessory** | Trailing **`Customize…`** button (standard small bordered button, same chrome as the "Manage" button on the model row). No chevron, no accent tint. |

- Tapping **`Customize…`** opens the editor sheet in **Global mode** (§4).
- The row (button + interaction) is **enabled only when AI Analysis & Summary is
  on**. When that toggle is off, the row is disabled/greyed, consistent with the
  other AI rows that depend on a model being present. (The subtitle text remains
  visible.)

### 3.2 Meeting detail → Re-summarize

On a meeting's detail view, the existing **`ellipsis.circle` overflow menu** already
has a **Regenerate Summary** item (enabled when the meeting has a transcript and a
model is downloaded). Its behavior changes:

- **Regenerate Summary** now opens the editor sheet in **Per-meeting mode** (§4)
  instead of regenerating immediately. The sheet is the deliberate confirmation
  surface; its primary button performs the regeneration.
- This replaces the previous "confirm overwrite" dialog for the regenerate path
  (the warning is folded into the sheet — see §6.3). First-time generation is
  unchanged (§3.3).

### 3.3 First-time "Generate Summary" is unchanged

The Summary tab's **Generate Summary** button (shown when a transcript exists but no
summary has been generated yet) keeps its current behavior: it generates directly
using the **saved** prompt, without opening the sheet. The sheet is for
*customizing*; first-run uses the saved/default prompt. (A user who wants to
customize the first run can generate, then Regenerate via the menu.)

## 4. The editor sheet

A single reusable SwiftUI sheet, presented with the standard `.sheet` modifier,
following existing Biscotti sheet conventions (native-simple — see UI design).
It has two modes that vary only the header copy, the footer primary action, and a
few mode-specific controls.

### 4.1 Shared structure (both modes)

Top → bottom:

1. **Header** — title + one-line subtitle (mode-specific copy below).
2. **Prompt editor** — the existing `MarkdownEditor` component (from
   `MarkdownEditorUI`), editable, bound to the working prompt text. It already
   themes Markdown (heading tinting, etc.); no custom syntax styling is added.
3. **Add example** affordance — lets the user append a pre-canned example block to
   the end of the prompt (§5).
4. **Footer** — `Restore default` (left) · spacer · mode-specific buttons (right).

### 4.2 Global mode (from Settings)

- **Title:** Summary Prompt
- **Subtitle:** These are the instructions Biscotti sends the on-device model to
  write each meeting summary. Edit them directly. Changes apply to every future
  summary.
- **Working text** initialized from the **effective saved prompt** (custom value if
  set, otherwise the factory default).
- **Footer primary:** **`Save`** (prominent, sage tint). **`Cancel`** to its left.
- **Save:** persists the working text to the global setting and dismisses. Takes
  effect on the next summary generated. No existing summaries change.
  **Clear-to-default rule:** if the working text (trimmed) equals the current
  factory default, the setting is **cleared** (stored empty = "use default")
  rather than storing the literal default text — so the user keeps tracking future
  default updates. This applies to every persist path (Global Save and per-meeting
  "Also save…").

### 4.3 Per-meeting mode (from a meeting's Regenerate Summary)

- **Title:** Re-summarize Meeting
- **Subtitle:** Customize the prompt for this meeting's summary, then regenerate.
  Your saved prompt stays as it is unless you choose to save these changes.
- **Working text** initialized from the **effective saved prompt** (the same value
  Global mode would show) — i.e. the user tweaks their normal prompt for this run.
- **Extra control:** a checkbox **`Also save these changes as my default`**
  (default **off**). When on, Save-on-regenerate also persists the working text to
  the global prompt.
- **Replace warning:** if this meeting's summary was **manually edited**
  (`editedSummary == true`), show a small inline caption in the sheet:
  "Regenerating will replace the summary you edited for this meeting." (No separate
  pre-sheet dialog.)
- **Footer primary:** **`Regenerate`** (prominent, sage tint). **`Cancel`** to its
  left.
- **Regenerate:** runs summary generation for **this meeting only**, using the
  working text as the summary instruction, against the meeting's existing
  transcript. Dismisses the sheet and switches to the Summary tab to show streaming
  output (matching today's regenerate UX). If the checkbox is on, the working text
  is also saved to the global prompt (subject to the clear-to-default rule, §4.2).
- **Result ownership (`editedSummary`):** a one-off custom prompt is real user
  effort, so the regenerated summary is protected like a hand-edited one. After the
  run, set `editedSummary = true` **iff the prompt actually used differs from the
  global prompt as it stands after this action's save step**. Concretely:
  - No edits (working text == effective global prompt) → `editedSummary = false`
    (plain regenerate; normal AI output).
  - Edited, checkbox **off** → used prompt ≠ global → `editedSummary = true`
    (meeting-specific, crafted result; auto-runs won't overwrite it and a later
    regenerate shows the replace warning).
  - Edited, checkbox **on** → the save makes the global prompt equal the used
    prompt → `editedSummary = false` (this is now their standard prompt, so the
    result is a standard result).

### 4.4 Restore default (both modes)

- The **`Restore default`** action replaces the **editor contents** with the factory
  default text (§2.2). It does **not** persist anything by itself — the change takes
  effect only via Save (Global) or Regenerate (Per-meeting), and only persists
  globally in the cases described above.
- Because it discards the current editor contents, it **confirms first** when the
  editor differs from the default (`.confirmationDialog`): "Restore the default
  summary prompt? Your current edits in this editor will be replaced." Confirm /
  Cancel. If the editor already equals the default, Restore is a no-op (may be
  disabled).

## 5. Example additions

The sheet offers a small set of pre-canned **example blocks** the user can append to
the end of the prompt, to spark ideas. Native-simple treatment (no decorative pill
chips with custom "added" states): an **`Add Example`** pull-down menu (or an
equivalent compact, discoverable control — finalized in UI design) listing the
examples by name. Choosing one **appends its block to the end** of the editor text,
separated by a blank line.

### 5.1 Behavior

- Appends to the **end** only (no insert-at-cursor).
- After appending, the editor scrolls to / reveals the appended text if practical.
- **Duplicate guard:** if an example's block is already present verbatim in the
  editor, that menu item is **disabled** (greyed), so the user can't append it
  twice. (No custom "added/check" chip styling.)
- Appending is a normal edit — it can be further edited or removed by hand, and is
  only persisted via Save/Regenerate like any other edit.

### 5.2 Example blocks

The same set is offered in both modes (one consistent list):

```
Slack recap →
## Notes for Slack
A short, paste-ready recap I can drop in our team channel — 4 lines max.

Meeting feedback →
## Meeting Feedback
3 things I did well, and 3 I could do better.

Decisions →
## Decisions
Every decision made, and who owns it.

Key quotes →
## Key Quotes
2–3 notable verbatim quotes.

Sentiment →
## Sentiment & Tone
A read on the mood and energy of the room.
```

(Exact wording can be refined during implementation; intent is short, useful,
self-contained "## Section" additions.)

## 6. Edge cases & error handling

### 6.1 Empty prompt

- A prompt that is empty or whitespace-only after trimming is invalid. **Save**
  (Global) and **Regenerate** (Per-meeting) are **disabled** while the prompt is
  empty, with a small inline caption: "The prompt can't be empty." `Restore default`
  remains available to recover.

### 6.2 Unsaved changes on Cancel

- If the working text differs from what it was when the sheet opened, **Cancel**
  (and other dismissals — Esc, clicking away if the sheet allows it) **confirm
  first** via `.confirmationDialog`: "Discard your changes?" → **Discard** /
  **Keep editing**. If there are no changes, Cancel dismisses immediately.

### 6.3 Regenerating over a manually edited summary

- Handled inline in Per-meeting mode (§4.3): the sheet shows the replace warning
  caption; pressing **Regenerate** proceeds (it is the explicit confirmation). The
  resulting `editedSummary` flag follows the result-ownership rule (§4.3): a plain
  or promoted-to-default regenerate leaves it `false`; an unsaved custom one-off
  sets it `true`. There is no separate force-regenerate confirm dialog.

### 6.4 AI Analysis & Summary turned off

- The Settings row is disabled (§3.1), so Global mode is unreachable from Settings.
- Per-meeting **Regenerate Summary** keeps its existing enablement (requires a
  transcript and a downloaded model) and is **not** gated by the auto-analysis
  toggle — manual intent always works, matching today's `runAnalysis` path.

### 6.5 No model / generation failure

- Regenerate surfaces failures through the **existing** summary-generation error
  states in the Summary tab (no new error UI). If no model is available, the
  Regenerate entry point is disabled exactly as today.

### 6.6 Very long prompts

- No hard length cap. Optionally, a subtle inline caption may appear past a large
  threshold (e.g. very long prompts may slow on-device generation). This is a
  low-priority nicety; if it complicates the editor, omit it. Never block on length.

### 6.7 Concurrent / in-flight generation

- Follow existing behavior: the Regenerate entry point reflects the current
  enhancement status (it is already disabled/guarded while a job runs for that
  meeting). Opening the sheet does not start a job; only the primary button does.

## 7. Data & persistence

- A new global setting holds the user's summary prompt. Empty/unset means "use the
  built-in factory default," so the shipped default can evolve for users who never
  customized. The editor always shows the **effective** prompt (custom if set, else
  default).
- The setting only ever holds a *genuinely custom* prompt. On any persist, if the
  working text (trimmed) equals the current factory default, the setting is
  **cleared** to empty rather than storing the literal default — keeping the user on
  the evolving built-in default (§4.2 clear-to-default rule). This also means
  saving after **Restore default** naturally clears the setting.
- The global prompt is persisted alongside other AI settings (SwiftData
  `AppSettings`). Changing it affects only **future** summaries.
- Per-meeting one-off edits are **not** persisted unless "Also save…" is checked.
- The generated summary is stored in the meeting's existing `summary` field
  (overwritten in place, as today — no new history/undo is introduced).

## 8. Generation wiring (functional level)

- The configured/effective summary prompt **substitutes for**
  `IntelligencePrompts.summaryTaskInstructions` at the two call sites that use it:
  the summary-only first-user turn and the summary follow-up turn. All other prompt
  construction (system, meeting details, transcript, speaker turn, title turn) is
  unchanged.
- Global summaries (auto-run after transcription, and the manual first-run/Generate)
  use the **saved** prompt.
- Per-meeting regenerate passes the working text as a **one-off override** for that
  single run; it is used instead of the saved prompt for that generation only.

## 9. Acceptance criteria

1. Settings shows a **Summary Prompt** row between AI Analysis & Summary and AI
   Language Model, with a `Customize…` button; the row is disabled when AI Analysis
   & Summary is off.
2. `Customize…` opens the editor sheet (Global mode) pre-filled with the effective
   prompt; **Save** persists it; later summaries use the new prompt; existing
   summaries are untouched.
3. With the default prompt unchanged, summary output is identical to today's
   behavior.
4. **Restore default** (with confirm) returns the editor to the factory default
   text; Save persists it; the setting tracks the built-in default thereafter.
5. **Add Example** appends a block to the end; a block already present has its menu
   item disabled.
6. Empty prompt disables Save/Regenerate with an inline note; Cancel with unsaved
   changes confirms before discarding.
7. A meeting's **Regenerate Summary** opens the sheet (Per-meeting mode) pre-filled
   with the saved prompt; **Regenerate** regenerates only that meeting using the
   working text and streams into the Summary tab; the saved prompt is unchanged
   unless **Also save these changes as my default** is checked.
8. Regenerating a manually edited summary shows the inline replace warning and, on
   Regenerate, overwrites it.
9. A per-meeting regenerate with an unsaved custom prompt marks the result
   `editedSummary = true`; a no-edit regenerate, or one with "Also save…" checked,
   leaves it `false`.
10. No template variables are surfaced anywhere; the editor edits plain instruction
    text only.
